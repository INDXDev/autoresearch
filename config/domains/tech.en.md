# Domain guide — Tech (English)

You are researching the run's topic through a **software / engineering /
product** lens: tools, infrastructure, protocols, open source, developer
practice.

## The three layers in this domain

1. **Foundations — the unwavering facts.** Stable specs, widely-adopted tools and
   protocols, durable architecture patterns, long-lived official docs. The things
   a practitioner can rely on.
2. **Latest — what's new.** Releases and major version bumps, RFCs, launches,
   notable new repos, incidents/postmortems — each with a real link you opened and
   an approximate date.
3. **Takes — ideas worth building.** Concrete tech ideas the foundations + the
   latest developments make possible *right now*: a tool, feature, integration,
   library, or architecture worth trying. State the idea as a clear proposal, the
   insight or gap behind it (why now), a first prototype or spike that would
   validate it, and the main risk. Adoption theses ("X is ready for production for
   use-case Y") count too.

## Preferred sources

WebFetch `config/priority_sources.md` first, then prefer: official docs and
changelogs, GitHub releases/issues, standards bodies (IETF RFCs, W3C, WHATWG),
reputable engineering blogs, and Hacker News for signal (verify the primary
source behind any HN link).

## What a good item looks like

Concrete and load-bearing for a builder: a version, a benchmark, a breaking
change, a real trade-off. Prefer primary sources (release notes, the repo, the
spec) over aggregated hot takes.

## Always

- Read `config/research_topics.md` for the lab's focus and constraints.
- NEVER invent releases, versions, repos, or URLs — include only what you found.
- Write all text fields in **English**.
